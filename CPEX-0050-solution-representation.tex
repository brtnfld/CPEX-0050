\section{Abstract}
%% ═════════════════════════════════════════════════════════════════════════

This CPEX proposes adding a \texttt{SolutionContinuity\_t}
enumeration and corresponding SIDS node to the CGNS standard, enabling
CGNS files to record whether solution data uses a continuous or
discontinuous field continuity.  This distinction---most commonly
associated with Continuous Galerkin (CG) and Discontinuous Galerkin
(DG) finite element methods---is fundamental to the correct
interpretation of solution data but is not captured by any existing
CGNS construct.  The proposal is general: it applies at any polynomial
order, to any discretization family (finite element, finite volume,
spectral element, etc.), and interacts naturally with the higher-order
solution infrastructure introduced by CPEX~0045.

%% ═════════════════════════════════════════════════════════════════════════
\section{Motivation and Rationale}
%% ═════════════════════════════════════════════════════════════════════════

\subsection{Problem Statement}

Modern CFD and computational mechanics codes increasingly use
discretization methods whose solution data cannot be fully described
by the existing CGNS metadata.  The core issue is the treatment of
degrees of freedom (DOFs) at shared geometric entities (vertices,
edges, faces):

\begin{itemize}[leftmargin=2em]
\item In a \textbf{continuous} representation (e.g., Continuous
  Galerkin FEM, $C^0$ spectral element methods), DOFs at shared
  interfaces are \emph{shared}: a single value exists per geometric
  node, and the global solution field is at least $C^0$-continuous
  across element boundaries.

\item In a \textbf{discontinuous} representation (e.g., Discontinuous
  Galerkin, finite volume, $C^{-1}$ spectral element methods), DOFs
  are \emph{element-local}: each element owns its own values at every
  point, including at shared interfaces.  Multiple independent values
  may exist at the same geometric location.
\end{itemize}

This distinction affects:

\begin{enumerate}[leftmargin=2em]
\item \textbf{Data size and layout.}  A DG solution on a mesh with
  shared vertices stores more values than a CG solution on the same
  mesh.  Post-processors that assume shared-vertex semantics will
  misinterpret the data dimensions.

\item \textbf{Visualization and interpolation.}  CG fields can be
  interpolated globally using the mesh connectivity; DG fields require
  element-local interpolation and may exhibit intentional
  discontinuities at element interfaces that should not be smoothed.

\item \textbf{Restart and data exchange.}  Solvers reading restart
  files must know whether DOFs are shared or element-local to
  correctly reconstruct the solution state.  Coupling frameworks
  exchanging solution data between codes need this metadata to perform
  correct inter-code transfers.

\item \textbf{Error estimation and adaptation.}  The jump in solution
  values across element interfaces in a DG solution is physically
  meaningful and is used for error estimation.  A post-processor that
  incorrectly treats DG data as continuous would average out these
  jumps.
\end{enumerate}

\subsection{Inadequacy of Existing CGNS Constructs}

No existing CGNS mechanism captures the continuous/discontinuous
distinction:

\begin{description}[leftmargin=2em]
\item[\texttt{GridLocation\_t}] specifies \emph{where} data is
  located (vertices, cell centers, faces, edges, or interpolation
  points) but not whether values at shared locations are shared or
  element-local.  A \texttt{Vertex}-located solution could be either
  CG or DG; the same applies to \texttt{InterpolationPoints}.

\item[\texttt{PointSet}] (\texttt{PointList}, \texttt{PointRange})
  can index subsets of data but does not encode continuity semantics.

\item[\texttt{InterpolationType\_t}] (CPEX~0045) describes the
  polynomial basis (e.g., Lagrange, monomial) but not whether
  that basis yields a globally continuous or element-local field.

\item[\texttt{SolutionInterpolation\_t}] (CPEX~0045) records
  polynomial order and basis type per element type but contains no
  continuity information.

\item[Data array size] can sometimes be used to \emph{infer} the
  representation---a DG field will have $N_{\mathrm{elem}} \times
  N_{\mathrm{DOF/elem}}$ values versus $N_{\mathrm{nodes}}$ for CG.
  However, this inference is fragile, ambiguous for $p=0$
  (cell-centered) data, and breaks entirely for mixed-element meshes
  or when point sets are used.
\end{description}

\subsection{Generality Beyond Higher-Order Methods}

Although often discussed in the context of high-order finite elements,
the continuous/discontinuous distinction is \textbf{not limited to
higher-order methods}:

\begin{itemize}[leftmargin=2em]
\item \textbf{DG $p=0$} (piecewise constant) is essentially a
  cell-centered finite volume representation.
\item \textbf{DG $p=1$} (piecewise linear, discontinuous) is widely
  used and is a low-order method.
\item \textbf{CG $p=1$} is standard continuous linear FEM, the most
  common finite element discretization.
\item Classical \textbf{finite volume} representations are inherently discontinuous ($p=0$ DG).
\item \textbf{Hybridizable DG (HDG)} methods have both element-local
  DOFs (discontinuous) and trace DOFs on faces (continuous on the
  skeleton).
\end{itemize}

A proper metadata field should therefore live at the
\texttt{FlowSolution\_t} level, applicable at any order, not solely
within the higher-order interpolation infrastructure.

%% ═════════════════════════════════════════════════════════════════════════
\section{Detailed Description}
%% ═════════════════════════════════════════════════════════════════════════

\subsection{Design Overview}

The proposal introduces:

\begin{enumerate}[leftmargin=2em]
\item A new \texttt{SolutionContinuity\_t} enumeration with values
  covering the common continuity types.
\item A new optional child node of \texttt{FlowSolution\_t},
  \texttt{DiscreteData\_t}, and \texttt{ZoneSubRegion\_t} recording
  the solution continuity.
\item Mid-Level Library functions to read and write the continuity.
\item Fortran bindings.
\end{enumerate}

\subsection{New Enumeration: \texttt{SolutionContinuity\_t}}
\label{sec:enum}

\begin{lstlisting}[language=C]
typedef enum {
  CGNS_ENUMV( SolutionContinuityNull )        = CG_Null,
  CGNS_ENUMV( SolutionContinuityUserDefined )  = CG_UserDefined,
  CGNS_ENUMV( Continuous )                          = 2,
  CGNS_ENUMV( Discontinuous )                       = 3
} CGNS_ENUMT( SolutionContinuity_t );

#define NofValidSolutionContinuity 4

extern CGNSDLL const char *
    SolutionContinuityName[NofValidSolutionContinuity];
\end{lstlisting}

The enumeration values are:

\begin{description}[leftmargin=2em]
\item[\texttt{SolutionContinuityNull}] No continuity
  specified.  This is the default for backward compatibility: existing
  files and codes that do not set this field are unaffected.

\item[\texttt{SolutionContinuityUserDefined}] Application-defined
  continuity not covered by the standard values.

\item[\texttt{Continuous}] The solution field is
  globally $C^0$-continuous (or higher) across element interfaces.
  DOFs at shared mesh entities (vertices, edges, faces) are shared
  between adjacent elements.  Typical of: Continuous Galerkin FEM,
  $C^0$ spectral element methods, isogeometric analysis with
  $C^0$~junctions.

\item[\texttt{Discontinuous}] The solution field is
  element-local with no continuity enforced across element interfaces
  ($C^{-1}$).  Each element owns independent DOFs, including at shared
  geometric locations.  Typical of: Discontinuous Galerkin methods,
  finite volume methods, $C^{-1}$ spectral element methods.
\end{description}

\subsubsection{Why Two Values Are Sufficient}

The \texttt{Continuous}/\texttt{Discontinuous} binary classification
is deliberate.  Methods that appear to require finer distinctions are
already handled by the existing two values combined with standard CGNS
patterns:

\begin{description}[leftmargin=2em]
\item[HDG and mixed-continuity methods] use both element-local
  volume DOFs and globally continuous trace DOFs.  These are stored as
  separate \texttt{FlowSolution\_t} nodes---one marked
  \texttt{Discontinuous}, the other \texttt{Continuous}---following the
  same pattern CGNS already uses to separate fields at different
  \texttt{GridLocation\_t} values.  A dedicated
  \texttt{MixedContinuity} value would conflate two distinct data
  layouts into a single node, making it harder, not easier, for
  post-processors to interpret the data.

\item[Penalized / weakly-continuous DG methods] enforce continuity
  through penalty terms in the variational formulation, but the data
  layout remains element-local: each element owns its own DOFs.  The
  correct label is \texttt{Discontinuous}.  ``Weak continuity'' is a
  solver-internal property of the numerical method, not a property of
  the stored data that readers and post-processors need to act on.

\item[Isogeometric analysis ($C^k$, $k \geq 1$)] uses basis functions
  with higher-than-$C^0$ smoothness, but the data layout is still
  globally continuous with shared DOFs at patch interfaces.  The
  correct label is \texttt{Continuous}.  The smoothness order $k$ is a
  property of the basis, not of the data layout; it belongs in the
  interpolation metadata (CPEX~0045), not in the continuity type.
\end{description}

The \texttt{SolutionContinuityUserDefined} value remains available
as an escape hatch, consistent with CGNS convention for all
enumeration types.

\subsection{SIDS Node Definition}
\label{sec:sids-node}

The continuity is stored as a new optional child node of
\texttt{FlowSolution\_t}.  The amended SIDS definition adds one line
(shown with the \texttt{+} marker).  The definition below includes the
\texttt{SolutionInterpolation\_t} child introduced by CPEX~0045:

\begin{sidsnode}{FlowSolution\_t (amended, SIDS Section 7.7)}
FlowSolution\_t< int CellDimension, int IndexDimension, \\
\qquad\qquad\quad int VertexSize[IndexDimension], \\
\qquad\qquad\quad int CellSize[IndexDimension] > := \\
\quad \{ \\
\quad List( Descriptor\_t Descriptor1 ... DescriptorN ) ;  \qquad\qquad (o) \\
\\
\quad GridLocation\_t GridLocation ; \qquad\qquad\qquad\qquad\qquad\quad\; (o/d) \\
\\
\quad Rind\_t<IndexDimension> Rind ; \qquad\qquad\qquad\qquad\qquad\quad (o/d) \\
\\
\quad IndexRange\_t<IndexDimension> PointRange ; \qquad\qquad\qquad\;\; (o) \\
\quad IndexArray\_t<IndexDimension, ListLength[], int> PointList ; \; (o) \\
\\
\quad List( DataArray\_t<DataType, IndexDimension, DataSize[]> \\
\qquad\quad DataArray1 ... DataArrayN ) ; \qquad\qquad\qquad\qquad\quad\;\; (o) \\
\\
\quad DataClass\_t DataClass ; \qquad\qquad\qquad\qquad\qquad\qquad\quad\; (o) \\
\\
\quad DimensionalUnits\_t DimensionalUnits ; \qquad\qquad\qquad\quad\;\;\; (o) \\
\\
\quad SolutionInterpolation\_t SolutionInterpolation ; \qquad\quad\;\;\;\; (o) \quad\textit{(CPEX 0045)} \\
\\
+ \quad \textcolor{red}{SolutionContinuity\_t SolutionContinuity} ; \qquad\quad\;\; (o) \\
\\
\quad List( UserDefinedData\_t UserDefinedData1 ... UserDefinedDataN ) ; (o) \\
\quad \}
\end{sidsnode}

The \texttt{SolutionContinuity} child is \textbf{optional}
(\texttt{o}).  When absent, the continuity is unspecified
(\texttt{SolutionContinuityNull}), preserving full backward
compatibility.

The same optional child node is added to \texttt{DiscreteData\_t}
(SIDS Section~12.4), using identical structure and semantics:

\begin{sidsnode}{DiscreteData\_t (amended, SIDS Section 12.4)}
DiscreteData\_t< int CellDimension, int IndexDimension, \\
\qquad\qquad\quad int VertexSize[IndexDimension], \\
\qquad\qquad\quad int CellSize[IndexDimension] > := \\
\quad \{ \\
\quad ... (existing children unchanged) ... \\
\\
+ \quad SolutionContinuity\_t SolutionContinuity ; \qquad\quad\;\; (o) \\
\quad \}
\end{sidsnode}

The optional child node is also added to \texttt{ZoneSubRegion\_t}
(SIDS Section~7.9).  \texttt{ZoneSubRegion\_t} holds
\texttt{DataArray\_t} nodes for localized solutions, boundary fluxes,
or fields on sub-regions where the continuous/discontinuous distinction
applies equally:

\begin{sidsnode}{ZoneSubRegion\_t (amended, SIDS Section 7.9)}
ZoneSubRegion\_t< int CellDimension, int IndexDimension, \\
\qquad\qquad\quad int VertexSize[IndexDimension], \\
\qquad\qquad\quad int CellSize[IndexDimension] > := \\
\quad \{ \\
\quad ... (existing children unchanged) ... \\
\\
+ \quad SolutionContinuity\_t SolutionContinuity ; \qquad\quad\;\; (o) \\
\quad \}
\end{sidsnode}

\subsubsection{Node Placement Rationale}

The node is placed as a child of \texttt{FlowSolution\_t} (rather
than, for example, \texttt{Zone\_t} or
\texttt{SolutionInterpolation\_t}) because:

\begin{itemize}[leftmargin=2em]
\item The continuity is a property of the \emph{solution field},
  not of the mesh or the zone.

\item A single zone may contain multiple \texttt{FlowSolution\_t}
  nodes with different continuity---for example, an HDG solver
  storing both volume (discontinuous) and trace (continuous) solutions,
  or a multi-physics code storing a continuous displacement field
  alongside a discontinuous stress field.

\item This parallels the existing \texttt{GridLocation\_t} child of
  \texttt{FlowSolution\_t}, which is also a per-solution-node
  property.

\item Placing it within the CPEX~0045 \texttt{SolutionInterpolation\_t}
  would incorrectly limit the scope to higher-order solutions, when
  the distinction applies at any order.
\end{itemize}

\textbf{Behavior:}

\begin{itemize}[leftmargin=2em]
\item The \texttt{\_write} functions create (or overwrite)
  the \texttt{SolutionContinuity} child node under the specified
  \texttt{FlowSolution\_t}, \texttt{DiscreteData\_t}, or
  \texttt{ZoneSubRegion\_t} node.  Writing
  \texttt{SolutionContinuityNull} removes the node if present.

\item The \texttt{\_read} functions return the value stored
  in the \texttt{SolutionContinuity} child node.  If the node does
  not exist (legacy file or unset), they return
  \texttt{SolutionContinuityNull} without error.
\end{itemize}

\subsection{Internal Data Structure Changes}

The \texttt{cgns\_sol} structure (\texttt{FlowSolution\_t}),
\texttt{cgns\_discrete} structure (\texttt{DiscreteData\_t}), and
\texttt{cgns\_subreg} structure (\texttt{ZoneSubRegion\_t}) in
\texttt{cgns\_header.h} each gain two fields:

\begin{lstlisting}[language=C]
typedef struct {            /* FlowSolution_t node */
    /* ... existing fields ... */

    /* CPEX 0050 */
    CGNS_ENUMT(SolutionContinuity_t) continuity;
    double continuity_id;   /* CGIO node id for overwrite/delete */
} cgns_sol;

typedef struct {            /* DiscreteData_t node */
    /* ... existing fields ... */

    /* CPEX 0050 */
    CGNS_ENUMT(SolutionContinuity_t) continuity;
    double continuity_id;
} cgns_discrete;

typedef struct {            /* ZoneSubRegion_t node */
    /* ... existing fields ... */

    /* CPEX 0050 */
    CGNS_ENUMT(SolutionContinuity_t) continuity;
    double continuity_id;
} cgns_subreg;
\end{lstlisting}

The \texttt{continuity} field defaults to
\texttt{SolutionContinuityNull} (value~0), which is the
zero-initialization value and the ``not specified'' state.  The
\texttt{continuity\_id} field stores the CGIO node identifier so
that the write function can overwrite or delete the child node; it
defaults to~0 (no node).

The string-to-enum mapping table must be initialized in

\begin{lstlisting}[language=C]
const char *SolutionContinuityName[NofValidSolutionContinuity] = {

    "UserDefined",

\end{lstlisting}

\subsection{MLL Implementation Requirements}
\label{sec:mll-impl}

\subsubsection{Internal Tree Parser Update (\texttt{cgns\_internal.c})}

When a CGNS file is opened, the MLL tree parser
(\texttt{cgi\_read\_node}) crawls the HDF5/CGIO tree and populates
internal C structures.  The parser must be updated to recognize child
nodes with label \texttt{SolutionContinuity\_t} during the
traversal of \texttt{FlowSolution\_t} (\texttt{cgi\_read\_sol}),
\texttt{DiscreteData\_t} (\texttt{cgi\_read\_discrete}), and
\texttt{ZoneSubRegion\_t} (\texttt{cgi\_read\_subreg}).

The required behavior is:

\begin{enumerate}[leftmargin=2em]
\item During memory allocation of \texttt{cgns\_sol},
  \texttt{cgns\_discrete}, and \texttt{cgns\_subreg}, the
  \texttt{continuity} field is zero-initialized, which corresponds
  to \texttt{SolutionContinuityNull}.
\item During the child-node traversal, if a child with label
  \texttt{"SolutionContinuity\_t"} is found, read its C1 string
  data and map it to the enum value via
  \texttt{cgi\_enum\_val("SolutionContinuity\_t", string\_data,
  \&enum\_val)}.  Store the result in the struct's
  \texttt{continuity} field.
\item If no such child node exists, the field retains its default
  \texttt{SolutionContinuityNull} value.
\end{enumerate}

This follows the existing pattern used for \texttt{GridLocation\_t}
parsing in \texttt{cgi\_read\_sol}.

\subsubsection{Read Function Error Handling}

The \texttt{\_read} functions (e.g.,
\texttt{cg\_sol\_continuity\_read}) return the cached struct field
rather than performing a fresh HDF5 read.  Because the internal tree
parser already handles the absent-node case by leaving the field at
\texttt{SolutionContinuityNull}, the read function simply returns
the struct value and \texttt{CG\_OK}:

\begin{lstlisting}[language=C]
int cg_sol_continuity_read(int fn, int B, int Z, int S,
                               CGNS_ENUMT(SolutionContinuity_t) *type) {
    cgns_sol *sol;
    cgns_file *cg;

    cg = cgi_get_file(fn);
    if (cg == 0) return CG_ERROR;

    if (cgi_check_mode(cg->filename, cg->mode, CG_MODE_READ))
        return CG_ERROR;

    sol = cgi_get_sol(cg, B, Z, S);
    if (sol == NULL) return CG_ERROR;

    *type = sol->continuity;

    return CG_OK;
}
\end{lstlisting}

This architecture avoids the need to intercept
\texttt{CG\_NODE\_NOT\_FOUND} errors at read time---the absent-node
case is handled once during file open, and all subsequent reads are
served from the in-memory cache.  Legacy code that iterates over
solutions and queries the continuity will always receive
\texttt{CG\_OK} with \texttt{SolutionContinuityNull} for files
that predate this CPEX.

\subsubsection{Write Function Implementation}

The \texttt{\_write} functions create (or overwrite) the
\texttt{SolutionContinuity} child node.  Writing
\texttt{SolutionContinuityNull} removes the node if present,
restoring the ``unspecified'' state.  The reference implementation for
\texttt{FlowSolution\_t} is shown below; the
\texttt{DiscreteData\_t} and \texttt{ZoneSubRegion\_t} variants are
analogous (substituting the appropriate internal structure accessor).

\begin{lstlisting}[language=C]
int cg_sol_continuity_write(int fn, int B, int Z, int S,
                                CGNS_ENUMT(SolutionContinuity_t) type) {
    cgns_sol *sol;
    cgns_file *cg;
    cgsize_t length;
    double posit_id;

    cg = cgi_get_file(fn);
    if (cg == 0) return CG_ERROR;

    if (cgi_check_mode(cg->filename, cg->mode, CG_MODE_WRITE))
        return CG_ERROR;

    sol = cgi_get_sol(cg, B, Z, S);
    if (sol == NULL) return CG_ERROR;

    /* Validate enum value */
    if (type < 0 || type >= NofValidSolutionContinuity) {
        cgi_error("Invalid SolutionContinuity value: %d", type);
        return CG_ERROR;
    }

    /* Delete existing node if writing Null */
    if (type == CGNS_ENUMV(SolutionContinuityNull)) {
        if (sol->continuity !=
                CGNS_ENUMV(SolutionContinuityNull)) {
            if (cgi_delete_node(sol->id,
                    sol->continuity_id))
                return CG_ERROR;
            sol->continuity =
                CGNS_ENUMV(SolutionContinuityNull);
            sol->continuity_id = 0;
        }
        return CG_OK;
    }

    /* Create or overwrite the child node */
    length = (cgsize_t)strlen(
        SolutionContinuityName[type]);
    if (cgi_new_node(sol->id, "SolutionContinuity",
            "SolutionContinuity_t", &posit_id,
            "C1", 1, &length,
            (void *)SolutionContinuityName[type]))
        return CG_ERROR;

    sol->continuity = type;
    sol->continuity_id = posit_id;
    return CG_OK;
}
\end{lstlisting}

\textbf{Note:} The \texttt{continuity\_id} field (a \texttt{double}
storing the CGIO node identifier) must be added to the internal
structures alongside the \texttt{continuity} field so that the
node can be located for overwrite or deletion.  This follows the
pattern used by other optional child nodes in the MLL (e.g.,
\texttt{GridLocation} tracking via \texttt{location\_id} fields in
some internal paths).

\subsection{Data Layout for Discontinuous Solutions at Shared Mesh Vertices}
\label{sec:shared-vertex-layout}

A common and important case arises when a DG solver uses a mesh whose
vertices are shared in the standard CGNS sense---enabling mesh walking,
stream tracing, and connectivity queries via the standard
\texttt{Elements\_t} tables---but stores element-local, independent
solution values at each element's copy of those vertices.  At a shared
vertex adjacent to $k$ elements there are $k$ distinct solution values,
one per element; readers must not average or conflate them.

The correct CGNS encoding for this pattern is
\texttt{GridLocation\_t = Vertex} combined with
\texttt{SolutionContinuity\_t = Discontinuous} and a
\texttt{PointList} that lists every element-vertex pair explicitly
(repeated vertex indices are expected and meaningful).  The
\texttt{SolutionContinuity\_t} node is the machine-readable flag that
tells readers these repeated entries represent independent DOFs, not
duplicate data.

Without the \texttt{SolutionContinuity\_t} node, a reader cannot
distinguish this layout from a malformed continuous solution with
duplicate entries, which is why the node is necessary.  With it, a
post-processor or coupling framework can correctly:

\begin{itemize}[leftmargin=2em]
\item walk the mesh using the shared-vertex connectivity for geometry
  operations (stream tracing, surface intersection, etc.);
\item read solution values element-locally, preserving the jumps at
  shared interfaces.
\end{itemize}

For element-local DOFs at non-vertex positions (interior quadrature
points, face quadrature points, etc.), use
\texttt{GridLocation\_t = InterpolationPoints} with
\texttt{SolutionInterpolation\_t} metadata from CPEX~0045.
\texttt{GridLocation\_t = Vertex} with a \texttt{PointList} is
specifically the correct choice when the DOF positions coincide with
mesh vertices and the shared-vertex mesh connectivity is to be
preserved.

\texttt{GridLocation\_t = Vertex} with \texttt{Discontinuous} and
\emph{no} \texttt{PointList} is not valid: the SIDS
\texttt{DataSize} formula for \texttt{Vertex} yields
$\texttt{DataSize} = \texttt{VertexSize}$ (the shared-vertex count),
which cannot represent the larger element-local DOF count.
\texttt{GridLocation\_t = CellCenter} with \texttt{Discontinuous}
remains valid since cell-centered data is inherently one value per
element with no shared-entity ambiguity.

See Example~\ref{ex:shared-vertex-dg} for the complete C encoding
of this pattern.

\section{Interaction with CPEX 0045 (Higher-Order Elements)}
%% ═════════════════════════════════════════════════════════════════════════

CPEX~0045 introduced \texttt{GridLocation\_t = InterpolationPoints},
\texttt{SolutionInterpolation\_t}, and the concept of per-element-type
polynomial basis definitions.  The present CPEX complements rather
than modifies that infrastructure:

\begin{itemize}[leftmargin=2em]
\item CPEX~0045 answers: ``What polynomial basis and order describes
  the solution within each element?''
\item This CPEX answers: ``Are the resulting DOFs shared across element
  interfaces or element-local?''
\end{itemize}

Both pieces of metadata are needed for a complete description of a
higher-order solution field.  They are orthogonal:

\begin{center}
\begin{tabular}{lcc}
\toprule
& \textbf{Continuous} & \textbf{Discontinuous} \\
\midrule
Lagrange $p=1$ & CG FEM & DG $p=1$ \\
Lagrange $p=3$ & CG spectral & DG spectral \\
Hierarchical $p=2$ & CG (modal) & DG (modal) \\
$p=0$ (constant) & --- & FV / DG $p=0$ \\
\bottomrule
\end{tabular}
\end{center}

For higher-order solutions using \texttt{GridLocation\_t =
InterpolationPoints}, the recommended practice is:

\begin{enumerate}[leftmargin=2em]
\item Use CPEX~0045 to define the element type, interpolation basis,
  and polynomial order via \texttt{SolutionInterpolation\_t}.
\item Use this CPEX to set \texttt{SolutionContinuity\_t} on the
  \texttt{FlowSolution\_t} node to indicate whether the field is
  continuous or discontinuous.
\end{enumerate}

For lower-order solutions using \texttt{GridLocation\_t = Vertex} or
\texttt{CellCenter}, this CPEX can be used independently of CPEX~0045.
See Section~\ref{sec:shared-vertex-layout} for the prescribed data
layout when DG DOFs coincide with shared mesh vertices.

\section{Backward Compatibility}
%% ═════════════════════════════════════════════════════════════════════════

\begin{itemize}[leftmargin=2em]
\item \textbf{Existing files:} Fully compatible.  The
  \texttt{SolutionContinuity} node is optional; its absence is
  equivalent to \texttt{SolutionContinuityNull}.  No existing file
  is invalidated.

\item \textbf{Existing readers:} Unaffected.  Per CGNS convention,
  readers silently ignore unknown child nodes.  An older library
  reading a file with \texttt{SolutionContinuity} nodes will
  simply skip them.

\item \textbf{Existing writers:} Unaffected.  Codes that do not call
  the new API functions produce files without
  \texttt{SolutionContinuity} nodes, identical to current
  behavior.

\item \textbf{ABI:} The \texttt{cgns\_sol}, \texttt{cgns\_discrete},
  and \texttt{cgns\_subreg} structures each grow by two fields (one
  \texttt{int}-sized enum and one \texttt{double} node identifier).
  This is an ABI change requiring recompilation of code that directly
  accesses the internal structures.  The public API is purely additive
  (new functions, new enum), so no existing function signatures change.
\end{itemize}

\subsection{Example 1: Writing a Continuous Galerkin Solution (C)}
\label{ex:cg-write}

\begin{lstlisting}[language=C]
/* Create a vertex-located solution node */
int S;
cg_sol_write(fn, B, Z, "FlowSolution", CGNS_ENUMV(Vertex), &S);

/* Mark as continuous (CG FEM) */
cg_sol_continuity_write(fn, B, Z, S,
    CGNS_ENUMV(Continuous));

/* Write solution data (one value per mesh vertex, shared) */
cg_field_write(fn, B, Z, S, CGNS_ENUMV(RealDouble),
    "Pressure", pressure, &F);
\end{lstlisting}

\subsection{Example 2: Writing a Discontinuous Galerkin Solution (C)}
\label{ex:dg-write}

\begin{lstlisting}[language=C]
/* Create an interpolation-point-located solution node */
int S;
cg_sol_write(fn, B, Z, "FlowSolution_DG",
    CGNS_ENUMV(InterpolationPoints), &S);

/* Set polynomial order (CPEX 0045) */
cg_sol_interpolation_order_write(fn, B, Z, S, 3, 0);

/* Mark as discontinuous */
cg_sol_continuity_write(fn, B, Z, S,
    CGNS_ENUMV(Discontinuous));

/* Write element-local DOFs */
cg_field_write(fn, B, Z, S, CGNS_ENUMV(RealDouble),
    "Pressure", pressure_dg, &F);
\end{lstlisting}

\subsection{Example 3: Reading and Dispatching on Continuity (C)}
\label{ex:dispatch}

\begin{lstlisting}[language=C]
CGNS_ENUMT(SolutionContinuity_t) repr;
cg_sol_continuity_read(fn, B, Z, S, &repr);

switch (repr) {
case CGNS_ENUMV(Continuous):
    /* Shared DOFs: interpolate using global mesh connectivity */
    visualize_continuous(fn, B, Z, S);
    break;
case CGNS_ENUMV(Discontinuous):
    /* Element-local DOFs: render per-element, preserve jumps */
    visualize_discontinuous(fn, B, Z, S);
    break;
default:
    /* Null or UserDefined: fall back to legacy heuristics */
    visualize_legacy(fn, B, Z, S);
    break;
}
\end{lstlisting}

\subsection{Example 4: HDG Solver with Both Continuity Types (C)}
\label{ex:hdg}

\begin{lstlisting}[language=C]
/* Volume DOFs (discontinuous) */
int S_vol;
cg_sol_write(fn, B, Z, "VolumeSolution",
    CGNS_ENUMV(InterpolationPoints), &S_vol);
cg_sol_continuity_write(fn, B, Z, S_vol,
    CGNS_ENUMV(Discontinuous));

/* Trace DOFs on element faces (continuous on skeleton).
 * Uses FaceCenter for p=0 traces (one DOF per face).
 * For higher-order traces, use InterpolationPoints
 * with CPEX 0045 metadata instead. */

cg_sol_write(fn, B, Z, "TraceSolution",
    CGNS_ENUMV(FaceCenter), &S_trace);
cg_sol_continuity_write(fn, B, Z, S_trace,
    CGNS_ENUMV(Continuous));
\end{lstlisting}

\label{ex:fortran}

\subsection{Example 6: Low-Order Finite Volume (DG $p=0$)}
\label{ex:fv}

\begin{lstlisting}[language=C]
/* Cell-centered FV solution */
int S;
cg_sol_write(fn, B, Z, "FlowSolution",
    CGNS_ENUMV(CellCenter), &S);

/* Explicitly mark as discontinuous */
cg_sol_continuity_write(fn, B, Z, S,
    CGNS_ENUMV(Discontinuous));
\end{lstlisting}

This example demonstrates that the continuity metadata is useful
even for traditional low-order methods.  While a \texttt{CellCenter}
solution is inherently element-local, explicitly recording the
continuity aids post-processors and coupling frameworks that need
to distinguish between solution types programmatically.

\subsection{Example 7: DG Solution on a Shared-Vertex Mesh (C)}
\label{ex:shared-vertex-dg}

This example addresses the case where the mesh uses shared vertices
(enabling stream tracing and mesh walking via standard
\texttt{Elements\_t} connectivity), but each element stores its own
independent solution values at those vertices.  The
\texttt{SolutionContinuity\_t} node is the flag that tells readers
the repeated vertex indices in the \texttt{PointList} are independent
DOFs, not duplicates.

\begin{lstlisting}[language=C]
/*
 * Mesh: nVerts shared vertices, nElems elements,
 *       nVertsPerElem vertices per element.
 * Solution: element-local DOFs at each element's vertices.
 *           Total DOFs = nElems * nVertsPerElem.
 */

/* Build the point list: for each element, list its vertex indices.
 * Shared vertices appear multiple times -- once per adjacent element. */
cgsize_t point_list[nElems * nVertsPerElem];
for (int e = 0; e < nElems; e++)
    for (int v = 0; v < nVertsPerElem; v++)
        point_list[e * nVertsPerElem + v] =
            element_connectivity[e][v];  /* 1-based vertex index */

/* Create solution node: Vertex location preserves shared-mesh
 * connectivity for geometry operations. */
int S;
cg_sol_write(fn, B, Z, "FlowSolution_DG",
    CGNS_ENUMV(Vertex), &S);

/* PointList defines the element-local DOF ordering.
 * DataSize = nElems * nVertsPerElem (not nVerts). */
cg_ptset_write(fn, B, Z, S,
    CGNS_ENUMV(PointList),
    (cgsize_t)(nElems * nVertsPerElem),
    point_list);

/* Mark as discontinuous: values at repeated vertex indices
 * are independent across elements. */
cg_sol_continuity_write(fn, B, Z, S,
    CGNS_ENUMV(Discontinuous));

/* Write element-local solution data */
cg_field_write(fn, B, Z, S, CGNS_ENUMV(RealDouble),
    "Pressure", pressure_dg, &F);
\end{lstlisting}

A reader distinguishes this from a continuous solution as follows:

\begin{lstlisting}[language=C]
CGNS_ENUMT(SolutionContinuity_t) cont;
cg_sol_continuity_read(fn, B, Z, S, &cont);

if (cont == CGNS_ENUMV(Discontinuous)) {
    /* PointList entries are element-local DOF indices.
     * Values at the same vertex index from different elements
     * are independent -- do not average. */
}
\end{lstlisting}

\section{File Mapping Impact}
%% ═════════════════════════════════════════════════════════════════════════

The \texttt{SolutionContinuity} child node is stored identically
to \texttt{GridLocation}: as a character node containing the string
name of the enumeration value.  The file-mapping attributes are:

\begin{center}
\begin{tabular}{@{} l l @{}}
\toprule
\textbf{Attribute} & \textbf{Value} \\
\midrule
Node name  & \texttt{SolutionContinuity} \\
Label      & \texttt{SolutionContinuity\_t} \\
Data type  & \texttt{C1} (character) \\
Dimension  & 1 \\
Dimension value & length of enumeration string \\
Data       & enumeration name string \\
\bottomrule
\end{tabular}
\end{center}

\noindent This follows the same pattern as the \texttt{GridLocation}
node under \texttt{FlowSolution\_t}: node name
\texttt{"GridLocation"}, label \texttt{"GridLocation\_t"}, data type
\texttt{"C1"}, with the enumeration string as data.

The node is written only when the continuity is not
\texttt{SolutionContinuityNull}.  When absent, readers default to
\texttt{SolutionContinuityNull}, paralleling how an absent
\texttt{GridLocation} node defaults to \texttt{Vertex}.

\section{Design Decisions}
%% ═════════════════════════════════════════════════════════════════════════

The following design questions were considered during the development
of this proposal.  Each is resolved below with rationale grounded in
existing CGNS conventions and SIDS precedent.

\begin{enumerate}[leftmargin=2em]

\item \textbf{Scope: \texttt{BCData\_t} is excluded.}
  \texttt{SolutionContinuity\_t} is \emph{not} added to
  \texttt{BCData\_t}.  Boundary condition data arrays in
  \texttt{BCData\_t} are sized by the associated
  \texttt{PointList}/\texttt{PointRange} and \texttt{GridLocation\_t},
  which already determine the data layout unambiguously.  In DG
  methods, boundary conditions are typically applied weakly through
  numerical fluxes; the element-local nature of the boundary data
  follows from the solution continuity on the parent
  \texttt{FlowSolution\_t} node.  Adding
  \texttt{SolutionContinuity\_t} to \texttt{BCData\_t} would be
  redundant and would introduce a potential inconsistency if the BC
  continuity differs from the solution it constrains.  Should a
  future use case require per-BC continuity metadata (e.g.,
  coupling frameworks imposing boundary data in a different
  continuity than the interior), it can be proposed as a
  separate, narrowly scoped CPEX without affecting this proposal.

\item \textbf{Per-solution-node granularity (not per-field).}
  The continuity applies at the \texttt{FlowSolution\_t} level,
  meaning all \texttt{DataArray\_t} children share the same
  continuity.  This follows the precedent set by
  \texttt{GridLocation\_t}, which is likewise a per-solution-node
  property: all fields within a \texttt{FlowSolution\_t} node share
  the same grid location.  Per-\texttt{DataArray\_t} granularity was
  rejected because it would add significant API and metadata
  complexity with no demonstrated need.  Solvers that genuinely mix
  continuity types within a single zone (e.g., HDG methods with
  continuous traces and discontinuous volumes) should use separate
  \texttt{FlowSolution\_t} nodes, exactly as they would use separate
  nodes for fields at different \texttt{GridLocation\_t} values.

\item \textbf{\texttt{CellCenter} + \texttt{Continuous}: permitted
  but semantically inconsistent.}
  The MLL does \emph{not} enforce consistency between
  \texttt{SolutionContinuity\_t} and \texttt{GridLocation\_t}.
  A \texttt{CellCenter} solution is inherently element-local, making
  \texttt{Continuous} semantically contradictory, but the library
  shall not reject such combinations.  This follows the established
  CGNS convention of being permissive in what is accepted: the
  standard documents correct usage; validation is delegated to
  application-level tools such as \texttt{cgnscheck}.

  \textbf{Convention:} \texttt{SolutionContinuity\_t = Continuous}
  with \texttt{GridLocation\_t = CellCenter} should not be used.
  \texttt{cgnscheck} should issue a warning when this combination is
  encountered.

\end{enumerate}
